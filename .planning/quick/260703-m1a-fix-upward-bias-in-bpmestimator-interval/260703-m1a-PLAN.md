Root cause (confirmed by simulation): `update()` computes `stepDelta` by differencing ROUNDED absolute step indices `absStep = round((timestamp - startTime)/stepDurMs)`. Near a rounding boundary a constant grid offset flips absStep ±1: the +1 flip yields an over-tempo sample that ENTERS the EWMA, while the compensating −1 flip yields `stepDelta < stepsPerBeat` and is DISCARDED by the sub-beat skip. Fast samples kept, slow samples censored → bias is only upward. Secondary bug: dense (every-step) playing makes every interval sub-beat, so the skip branch advances `last*` every hit and NO sample is ever produced (BPM stays null).

  <name>Task 1: Rewrite BPMEstimator.update to use interval rounding and remove censoring bias</name>
  <files>src/midi/BPMEstimator.ts, src/midi/PerformanceTracker.ts</files>
  <behavior>
    - Constant grid offset cancels: two consecutive hits 500ms apart at division 16 (stepDurMs=125) always give stepDelta=round(500/125)=4 and bpmSample=120, regardless of any fixed offset applied to both timestamps.
    - Sub-beat accepted: hits 250ms apart at division 8 (stepDurMs=250, stepsPerBeat=2) give stepDelta=1, bpmSample=1*60/(2*0.25)=120 (previously discarded → null forever).
    - Flam skipped: interval that rounds to 0 steps (stepDelta < 1) OR interval < 100ms → skip WITHOUT advancing last* (next real onset measures from the true previous hit).
    - Minimum-interval floor: intervals < 100ms produce no sample (excludes noisy micro-intervals / sub-16th spacing at typical tempos).
  </behavior>
  <action>
Rewrite `update()` in src/midi/BPMEstimator.ts:

(1) Compute stepDelta from the INTERVAL, not from differenced rounded absolute steps: after seeding `lastTimestamp` on the first hit, use `const stepDelta = Math.round((timestamp - this.lastTimestamp) / stepDurMs);`. Numerator and denominator are now consistent, so any constant grid offset cancels exactly.

(2) Drop the `absStep` / `lastAbsStep` machinery entirely — it is the source of the bias. Track only `lastTimestamp` for interval math (and `bpmEstimate`). Remove the `lastAbsStep` field and its resets in `reset()`.

(3) Guard order in update():
   - Compute `const timeDelta = timestamp - this.lastTimestamp;` first.
   - Flam / minimum-interval floor: `if (timeDelta < 100) return;` — skip WITHOUT touching `lastTimestamp`, preserving flam semantics (the next genuine onset measures from the real previous hit). This single floor also covers the old `stepDelta < 1` (round-to-0) flam case since any interval that rounds to 0 steps at a sane tempo is well under 100ms; keep the comment explaining the rationale so it is not mistaken for censoring.
   - Then compute `stepDelta` and, defensively, `if (stepDelta < 1) return;` (still without advancing last*).
   - REMOVE the `if (stepDelta < stepsPerBeat)` sub-beat censoring branch — this is what caused the upward bias. Accept `stepDelta >= 1`.

(4) Compute `bpmSample = (stepDelta * 60) / (stepsPerBeat * timeDeltaSecs)` where `timeDeltaSecs = timeDelta / 1000`, feed the EWMA (seed on first sample, else `bpmEstimate += EWMA_ALPHA * (bpmSample - bpmEstimate)`), then set `this.lastTimestamp = timestamp;`.

(5) Debug logging (project convention: `logger.log()` is debug-only): after computing a real sample, `logger.log('[BPMEstimator]', { bpmSample, stepDelta, timeDeltaMs: timeDelta });`. Import `logger` from '../utils/logger'. Keep it a single cheap object log — no loops, no allocation-heavy work — the MIDI path must stay <1ms per event.

(6) Update the `startTime` parameter: it is now unused by the delta math and TypeScript strict `noUnusedParameters` will error. Drop it from the `update()` signature (new signature: `update(timestamp: number, tempo: number, division: number): void`) and update the single caller in src/midi/PerformanceTracker.ts:260 from `this.bpmEstimator.update(timestamp, this.startTime!, this.tempo, this.division)` to `this.bpmEstimator.update(timestamp, this.tempo, this.division)`. Do not change the `absError < 100` gate around it.

(7) Rewrite the class docstring: explain that stepDelta is computed by rounding the inter-onset INTERVAL so numerator/denominator stay consistent and a constant grid offset cancels completely; note that the previous `stepDelta < stepsPerBeat` censoring caused an UPWARD bias (kept fast-looking boundary flips, discarded slow-looking ones) and has been removed; document the 100ms minimum-interval floor and its tradeoff (excludes sub-16th intervals at high tempo). Keep explicit return types on all methods (strict mode).
