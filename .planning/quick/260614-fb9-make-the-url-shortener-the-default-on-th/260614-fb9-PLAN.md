<objective>
Make the URL shortener the default for the QR tab in ShareModal. When the QR tab becomes active, automatically shorten the shareable URL and encode the resulting short URL into the QR code. Fall back to the long URL (including the existing "URL too long" wall) only if shortening fails or the shortener is not configured.

Purpose: Fixes the dead-end where long groove URLs exceed QR capacity (~2000 chars) and render an unusable "URL too long" wall. Short URLs always fit, so most grooves become scannable.

Output: Updated `ShareModal.tsx` with QR auto-shorten + loading + fallback, and extended `ShareModal.test.tsx`.
</objective>

  <files>src/components/production/ShareModal.tsx</files>
  <behavior>
    - When QR tab is NOT active, no shorten request is issued.
    - When QR tab becomes active and shortener is configured, shortenURL(shareableURL) is called exactly once for that URL.
    - While shortening, a loading state is exposed so the QR area can show a spinner.
    - On success, the QR encodes the returned short URL.
    - On failure or when shortener is not configured, the QR encodes the long URL (qrURL === shareableURL).
    - When shareableURL changes (urlMode toggle) while QR is active, a shorten attempt re-runs for the new URL; the per-session cache prevents duplicate network calls for already-seen URLs.
  </behavior>
  <action>
    Add QR-specific shorten state, separate from the Link tab state so the Link tab cannot regress: introduce `qrShortURL: string | null` and `isShorteningQR: boolean` via useState (named state, explicit types; use null not undefined for "no value" per CLAUDE.md conventions). Do NOT reuse `shortURL`/`isShortening` for the QR flow — those belong to the manual Link tab.

    Add a useEffect keyed on `[activeTab, shareableURL]` that: returns early unless `activeTab === 'qr'`; returns early (leaving qrShortURL null) when `isShortenerConfigured()` is false; otherwise sets `isShorteningQR` true, awaits `shortenURL(shareableURL)`, and on success stores the result in `qrShortURL`. On thrown error, leave `qrShortURL` null so the render falls back to the long URL (do not surface a blocking error in the QR area — the long-URL path with its existing too-long wall is the graceful fallback). Always clear `isShorteningQR` in a finally. Guard against stale async updates: capture the `shareableURL` the effect ran with and ignore the resolved value if `shareableURL` changed before the promise resolved (standard cancelled-flag pattern inside the effect).

    Reset QR state when the shareable URL changes: extend the existing `useEffect(..., [shareableURL])` (the one that resets `shortURL`/`shortenError`) to also reset `qrShortURL` to null so a urlMode toggle does not show a stale short URL before the new one resolves.

    Compute the value the QR encodes: `const qrURL = qrShortURL ?? shareableURL;`. Recompute `isURLTooLongForQR` against `qrURL` (not `shareableURL`) so a successfully shortened URL never trips the too-long wall, while the long-URL fallback still does. Keep `MAX_QR_URL_LENGTH = 2000`.

    Rewrite `renderQRTab()` to: (a) while `isShorteningQR` is true, render a centered loading state in the QR area (reuse the existing `Loader2` icon with `animate-spin`, matching the Link tab's spinner styling) instead of the QR; (b) otherwise render the existing too-long wall when `isURLTooLongForQR` (its copy already points users to the Link tab — keep it as the fallback for URLs that could not be shortened); (c) otherwise render `<QRCodeSVG value={qrURL} ... />` with the existing size/level/margin props.

    Analytics: when the QR auto-shorten succeeds, emit `trackShareMethod('shorten', { urlMode })` once per successful shorten (mirror the Link tab's existing call). Do not emit on cache hits that you cannot distinguish — calling it on each successful resolve is acceptable and matches existing behavior.

    Do not touch `handleShortenURL`, `renderLinkTab`, or the Link tab state. Implements QR-SHORTEN-01.
